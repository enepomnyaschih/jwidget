jwidget/Router

Consumption

import Router from "jwidget/Router";

Default export#

Hierarchy#

  • interface Destroyable
  • interface IClass
  • class Class
  • class Router<T extends Destroyable>

Description#

T
(extends Destroyable) Target object type.

URL router. Converts incoming part of URL (hash) to a target object and passes tail string to it for further routing. Target object may have any type, but usually this is Component. For simplicity, Router<Component> has a shorthand - UIRouter.

Terminology#
  • Path is some final part of URL (hash) relevant to a certain router. Path never contains "#" symbol.
  • Root path is URL (hash) without "#" symbol.
  • Route or route name is the first token of the path which is relevant to the next single step of the routing.
  • Argument is the remaining path after route extraction.

For example, hash value #inbox/3?filter=10 provides us with inbox/3?filter=10 root path, and it may get processed in 2 steps with the following values:

Path Route Argument
inbox/3?filter=10 inbox 3?filter=10
3?filter=10 3 ?filter=10
How it works#

Router takes an incoming string Property (for example, hash), parses it and provides an outcoming Property<T>. Outcoming property may contain any object you want. You specify this object in a callback function. This can be either general-purpose Handler function or a part of HandlerConfig<T> object.

In this example, hash is passed to Application's router object, and it builds a target property containing a child component to render by the following rules:

  • inbox/* path is mapped to Inbox component
  • compose/* path is mapped to Compose component
  • settings/* path is mapped to Settings component
  • blank path is automatically redirected to inbox
  • any other path is mapped to NotFound component

The following code demonstrates the common pattern of root router initialization:

protected beforeRender() {
    super.beforeRender();
    this.router = this.own(new UIRouter({
        path: hash,
        handler: {
            routes: {
                "inbox"   : arg => new Inbox(arg, this.router),
                "compose" : () => new Compose(),
                "settings": () => new Settings(),
                ""        : () => new RouteRedirector("inbox", this.router)
            },
            notFound: route => new NotFound(route)
        }
    }));
    this.router.update();
}

protected renderPage() {
    return this.router.target;
}

Notice that update method is called separately after router construction. It is implemented so to make sure that this.router field is assigned before routing. Sometimes it is a crucial factor (e.g. in the example above, this.router is used in "inbox" route handler).

Route hierarchy#

In the example above, Inbox provides further routing for all inbox/* paths based on its own rules. Applacation router passes path tail string to Inbox constructor to make this happen. Inbox rules:

  • inbox path is mapped to EmailList component
  • inbox/:id path is mapped to Email component if an email with such ID exists
  • inbox/:id path is mapped to EmailNotFound component if there's no email with such ID
protected beforeRender() {
    super.beforeRender();
    this.router = this.own(new UIRouter({
        name: "inbox",
        parent: this.parentRouter,
        path: this.path,
        handler: id => {
            if (!id) {
                return new EmailList(this.emails);
            }
            const email = this.emails.find(email => email.id === id);
            return email != null ? new Email(email, this.router) : new EmailNotFound(id);
        }
    }));
    this.router.update();
}

protected renderContent() {
    return this.router.target;
}

We can recognize the next differences between the root router and its child:

  • path value is different: the root router is based on full hash, and the child router is based on its remainder after "inbox/" substring.
  • Child router requires proper name to be specified. Router name is a chunk of the path that caused this route to get initialized. In our case, this is obviously "inbox". It is not certainly a constant: if we want "inbox/:id" route to have subroutes, then a specific email ID would be a name of the email router.
  • Child router requires proper parent to be specified. Pass parent router here. Every router should have a reference to its parent. It provides getFullPath and redirectRoute with a clue about all parts of the path. If your router provides you with wrong paths, check name and parent of all routers in your hierarchy - they are likely assigned to wrong values.
Routing flow#

Routing is performed in three steps:

  1. Incoming path string is parsed using separator callback into two tokens: route and argument. The route should be used to process this single routing step, and the argument should be passed to the target for further routing. Make sure that separator never returns two routes which have the same target. For example, if both "" and "inbox" lead to Inbox component, make sure that separator function returns the same route for them, for example, "index".

    this.router = this.own(new UIRouter({
    path: hash,
    separator: path => path ? Router.makeSeparator()(path) : ["inbox", null], // mirror "" to "inbox"
    handler: {
        routes: {
            "inbox"   : arg => new Inbox(arg, this.router),
            "compose" : () => new Compose(),
            "settings": () => new Settings()
        },
        notFound: route => new NotFound(route)
    }
}));

    Otherwise, expect your target component to be recreated when user switches back and forth between "" and "inbox". If separator function returns null or undefined, it is automatically mapped to ["", null]. Separator can be specified as a string. In this case, it is passed to makeSeparator function - see it for more details.

  2. If the route returned by separator is changed, the next steps are following:
    • Null is assigned to target property and the previous target gets destroyed
    • arg and route get assigned to new values
    • handler function is called to build a new target
    • Result is assigned to target property
  3. Else only arg gets reassigned to pass the control flow to the child routers.

Please notice that Router controls the life time of your target objects. For this reason, any target must implement Destroyable interface. You should also control the life time of your router (i.e. own it in the component).

Redirection#

2nd and 3rd steps above result in creation of many new objects that may cause unexpected redirections. This would make things fall apart. That's why jWidget router recommends to do such redirections asynchronously. If your component happen to do the redirection immediately and synchronously, the router will discard to handle it and throw an error.

However, outside routing cycle, you are free to do the redirections synchronously at any moment. The easiest way to do this is to use redirectRoute function. It takes a router as an optional argument. If you dismiss it, it will result in global redirection.

redirectRoute("inbox", router); // redirection in this router
redirectRoute("inbox"); // global redirection

To do the asynchronous redirection in a router handler function, render RouteRedirector as a target component.

this.router = this.own(new UIRouter({
    path: hash,
    handler: {
        routes: {
            "inbox": arg => new Inbox(arg)
        },
        notFound: () => new RouteRedirector("inbox", this.router)
    }
}));

This router will forcibly redirect any route to "inbox".

If you define custom separator function, you must also define an opposite joiner function for redirections to work properly.

Menu status management#

In order to highlight the active item of your menu, you can use route property.

const activeElement = this.router.route.map(route => el.filter('[data-route="' + route + '"]'));
new Switcher(activeElement, {
    init: el => el.css("font-weight", "bold"),
    done: el => el.css("font-weight", "")
});

Router configuration.

Creates router instance. Please notice that the router doesn't process current route immediately on initialization. To process the route, call update method.

Fields#

name#

readonly name: string

Router name. Must be equal to the route name in the parent router. Required for proper getFullPath and redirectRoute function processing. Root router does not have a name.

parent#

readonly parent: Router<any>

Parent router. Required for proper getFullPath and redirectRoute function processing. Root router does not have a parent.

path#

readonly path: Bindable<string>

Path that the router is bound to. Path is a final part of URL (hash) relevant to this router. Any path change results in update method call.

separator#

Path separator function used by the router. See separator for details.

joiner#

Path joiner function used by the router. See joiner for details.

handler#

Route handler function used by the router. See handler for details.

scope#

readonly scope: any

separator, joiner and handler call scope.

target#

readonly target: Bindable<T>

Router target. Main purpose of the router is to convert path to target. In particular, UIRouter creates Component instances based on current path value so you could render them.

route#

readonly route: Bindable<string>

Current route. First chunk of the path detected by separator function. You can watch this property to activate and deactivate items in your menu. See Menu status management.

arg#

readonly arg: Bindable<string>

Remainder of current route after separator function call. This property is passed to handler function and can be passed over to child components for further routing.

destroy (inherited from Class)#

(): void

Class destructor invocation method. Destroys all aggregated objects and calls destroyObject method. You must call this method explicitly from outside, because JavaScript doesn't support automatic class destructor calling.

const object = new MyClass();

// ...

// When the object is not needed anymore, destroy it.
object.destroy();

Alternatively (and optimally), you should use own method to aggregate this object inside another one.

destroyObject (inherited from Class)#

protected (): void

Class destructor implementation. Called in destroy method after destruction of all aggregated objects. The logic of class instance destruction should usually be implemented here. If you override this method, remember to call super.destroyObject() at the end of the method:

destroyObject() {
    // Release resources
    ...

    // Call superclass destructor
    super.destroyObject();
}

Aggregates the object. It means that the specified object is automatically destroyed on this object destruction. The aggregated objects are destroyed in reverse order. Returns obj object, which makes it easy to use in field definition:

The same as own, but returns this, which makes it easy to use in object instantiation:

const items = new BindableArray().ownValues();

Static fields#

DEFAULT_SEPARATOR#

DEFAULT_SEPARATOR: RegExp

Default value of separator: /^\/*([^?\/]+)(?:\/(.*)|(\?.*))?$/

Briefly, it follows the next logic:

  • Trims leading /
  • Looks for the first / or ?
  • Uses first part as route
  • Uses second part as argument. Questionmark ? is included to the argument, slash / is not

Examples:

Incoming pathResulting routeResulting argument
"" or null""null
"inbox""inbox"null
"inbox/""inbox"""
"inbox/1""inbox""1"
"inbox/1/edit""inbox""1/edit"
"/inbox""inbox"null
"/inbox/""inbox"""
"///inbox///""inbox""//"
"inbox?id=1""inbox""?id=1"
DEFAULT_JOINER#

DEFAULT_JOINER: string

Default value of joiner: "/"

Simply joins route and argument with "/"

Static methods#

makeSeparator#

(separator: Router.Separator | RegExp = Router.DEFAULT_SEPARATOR): Router.Separator

separator
Function or regular expression.
returns
Separator function.

If separator is a function, returns it immediately. Else converts the specified regular expression to a function by the following rule: The first token ($1) of path is used as a route, and the next non-null token ($2 or further) is used as an argument. If path is null, it is assumed to be "".

makeJoiner#

(joiner: Router.Joiner | string = Router.DEFAULT_JOINER): Router.Joiner

joiner
Function or separation character.
returns
Joiner function.

If joiner is a function, returns it immediately. Else converts the specified string to a function by the following rule: joins incoming route/argument pair via the specified string. Leading joiner symbols in argument are trimmed. If argument starts with "?", joiner symbol is not added. If argument is null or blank, returns route. Examples:

Incoming routeIncoming argumentSeparatorResulting path
"""""/"""
"inbox""""/""inbox"
"inbox""1""/""inbox/1"
"inbox""1/reply""/""inbox/1/reply"
"inbox""/1/reply""/""inbox/1/reply"
"inbox""/1/reply/""/""inbox/1/reply/"
"inbox""///1/reply///""/""inbox/1/reply///"
"inbox""?id=1""/""inbox?id=1"

Router.Separator#

(path: string): string[]

Signature of separator function. The function splits path to route and argument. Therefore, it must return two string values. If function returns null, it is assumed to be ["", null].

Router.Joiner#

(route: string, arg: string): string

Signature of joiner function. The function joins route and argument to a path.

Router.Handler#

T
Target object type.

(route: string, arg: Bindable): T

Signature of handler general-purpose function. The function maps the specified route to a target object (usually, Component) and passes argument to it for further routing.

Router.Route#

T
Target object type.

(arg: Bindable): T

Signature of a single route in handler object. The function maps a single route to a target object (usually, Component) and passes argument to it for further routing.

Router.HandlerConfig#

T
Target object type.

Router handler configuration object.

routes#

readonly routes?: { readonly [key: string]: Router.Route<T>; }

Map of specific route handlers. If current route is present in this dictionary, the router calls its corresponding handler and passes argument to it. Route and argument themselves are computed with separator callback.

notFound#

readonly notFound?: Router.Handler<T>

If none of the routes matches current route, the router calls this handler callback and passes both route and argument to it. By default, returns null for any input.

Router.Config#

T
Target object type.

Router configuration object.

name#

readonly name?: string

Router name. Router name is a chunk of the path that caused this route to get initialized. Root router doesn't have a name. See Route hierarchy for details.

parent#

readonly parent?: string

Parent router. It provides getFullPath and redirectRoute with a clue about all parts of the path. If your router provides you with wrong paths, check name and parent of all routers in your hierarchy - they are likely assigned to wrong values. Root router doesn't have a parent. See Route hierarchy for details.

path#

readonly path?: Bindable<string>

Path to bind the router to. Root router should usually get bound to hash property. Child routers should receive path from their parents.

target#

readonly target?: IProperty<T>

Target property. Router puts the result of handler function call to target property. If target is omitted, the router creates it automatically. Router automatically controls the life time of your targets, so, if you pass your precreated target property to a Router, make sure that it is not aggregating its value, i.e. ownValue method is not called.

separator#

readonly separator?: Router.Separator | RegExp

Path separator function. Parses incoming path to two tokens: route and argument. Route gets used to process a single routing step and create a target, argument gets passed to the target for further routing.

Separator can be specified as a function or a regular expression. The function has the next signature:

(path: string): string[]

It must return two string values. If it returns null, it is assumed to be ["", null].

Make sure that separator never returns two routes which have the same target. For example, if both "" and "inbox" lead to Inbox component, make sure that separator function returns the same route for them, for example, "inbox". Otherwise, expect your target component to be recreated when user switches back and forth between "" and "inbox".

If you define custom separator function, you must also define an opposite joiner function for redirections to work properly.

If the separator is specified as a regular expression, it gets converted to a function with Router.makeSeparator function - see it for more details.

Defaults to Router.DEFAULT_SEPARATOR. See it for more details.

joiner#

readonly joiner?: Router.Joiner | string

Path joiner. Opposite to separator. Used in getFullPath and redirectRoute methods to properly build the path. Joins incoming route and argument to a full path.

Joiner can be specified as a function or a string. The function has the next signature:

(route: string, arg: string): string

If the joiner is specified as a string, it gets converted to a function with Router.makeJoiner function - see it for more details.

Defaults to Router.DEFAULT_JOINER = "/".

handler#

readonly handler?: Router.Handler<T> | Router.HandlerConfig<T>

Route handler. Maps the route string to a target object and passes argument to it for further routing.

handler: (route, arg) => {
    const doc = this.docs[route];
    return doc ? doc.createView(arg) : new Page404();
}

Handler function must create a new object instead of returning an existing object, because Router controls life time of the target objects and destroys them on route leave.

Handler may return null or undefined if nothing needs to be assigned as a target - router will recognize this case and won't try to destroy anything.

Handler can be specified as an object matching Router.HandlerConfig interface. In this case, it gets converted to a function with Router.makeHandler function. See Router.HandlerConfig for more details.

Default handler function maps all routes to null.

scope#

readonly scope?: Router.Handler<T> | Router.HandlerConfig<T>

separator, joiner and handler call scope.
